# Wails v3 Internal Service Binding

## 🎉 Discovery: Wails v3 Can Bind `internal/` Packages!

### What We Learned

Previously, we thought we needed wrapper services in the `main` package to expose `internal/` services to the frontend. **This was wrong!**

Wails v3 **automatically generates TypeScript bindings** for services in `internal/` packages, including:
- ✅ Automatic `context.Context` handling
- ✅ Type conversion (Go → TypeScript)
- ✅ Full method exposure

### Before (Incorrect Approach)

```
internal/services/vector_search.go (core logic)
         ↓
vectorSearchService.go (wrapper in main package)
         ↓
frontend/bindings/.../vectorsearchservice.ts
```

**Problems:**
- ❌ Unnecessary wrapper code
- ❌ Duplicate method definitions
- ❌ Extra maintenance burden

### After (Correct Approach)

```
internal/services/vector_search.go (core logic)
         ↓
frontend/bindings/.../internal/services/vectorsearchservice.ts
```

**Benefits:**
- ✅ Direct binding to internal service
- ✅ No wrapper needed
- ✅ Cleaner architecture

## Implementation

### 1. Register Internal Service Directly

**main.go:**
```go
// Initialize Vector Search service
vectorSearchService, err := services.NewVectorSearchService(vectorDBPath, "llama", "http://localhost:8080")
if err != nil {
    log.Printf("⚠️  Warning: Failed to initialize Vector Search service: %v", err)
} else {
    log.Printf("✅ Vector Search service initialized (chromem)")
}

// Register directly - no wrapper needed!
app := application.New(application.Options{
    Services: []application.Service{
        application.NewService(vectorSearchService),
        // ... other services
    },
})
```

### 2. Use Generated Bindings in Frontend

**frontend/src/services/rag/client.ts:**
```typescript
// Import from internal/services binding (auto-generated by Wails)
const { SemanticSearch } = await import(
  '@@/github.com/kawai-network/veridium/internal/services/vectorsearchservice'
);

// Call directly - context is handled automatically!
const results = await SemanticSearch(
  this.userId,
  query,
  fileIds || [],
  30, // limit
);
```

### 3. Generated Binding Example

**frontend/bindings/.../internal/services/vectorsearchservice.ts:**
```typescript
/**
 * SemanticSearch performs semantic search on chunks
 */
export function SemanticSearch(
  userID: string, 
  query: string, 
  fileIDs: string[], 
  limit: number
): $CancellablePromise<$models.SearchResult[]> {
    return $Call.ByID(3829428111, userID, query, fileIDs, limit).then(($result: any) => {
        return $$createType6($result);
    });
}
```

**Note:** The Go method signature has `ctx context.Context` as the first parameter, but Wails **automatically removes it** from the TypeScript binding!

## How Wails v3 Handles Context

### Go Service Method

```go
// internal/services/vector_search.go
func (s *VectorSearchService) SemanticSearch(
    ctx context.Context,  // ← This parameter
    userID string,
    query string,
    fileIDs []string,
    limit int,
) ([]SearchResult, error) {
    // ...
}
```

### Generated TypeScript Binding

```typescript
// frontend/bindings/.../vectorsearchservice.ts
export function SemanticSearch(
    // ← context.Context is automatically removed!
    userID: string,
    query: string,
    fileIDs: string[],
    limit: number
): $CancellablePromise<$models.SearchResult[]>
```

### What Wails Does Behind the Scenes

1. **Detects `context.Context`** as the first parameter
2. **Automatically injects** `context.Background()` when calling from frontend
3. **Removes it** from TypeScript binding signature
4. **Handles cancellation** via `$CancellablePromise`

## When Do You Need a Wrapper?

You **only** need a wrapper service in `main` package if:

1. **Type Conversion Required**
   - Go types that don't map well to TypeScript
   - Need custom JSON marshaling

2. **Additional Logic Required**
   - Pre-processing before calling internal service
   - Post-processing results
   - Combining multiple internal services

3. **Security/Validation**
   - Need to validate user permissions
   - Sanitize inputs before passing to internal service

4. **Backwards Compatibility**
   - Maintaining old API while migrating to new internal structure

## Best Practices

### ✅ DO:
- Register `internal/` services directly when possible
- Let Wails handle context automatically
- Use generated bindings in frontend
- Keep business logic in `internal/` packages

### ❌ DON'T:
- Create unnecessary wrappers in `main` package
- Manually handle `context.Context` in frontend
- Duplicate method definitions
- Mix business logic with Wails bindings

## Migration Guide

If you have existing wrapper services:

1. **Check if wrapper is needed** (see "When Do You Need a Wrapper?" above)
2. **If not needed:**
   - Delete wrapper file (e.g., `vectorSearchService.go`)
   - Update `main.go` to register internal service directly
   - Update frontend imports to use `internal/services/...` path
   - Run `make generate` to regenerate bindings
   - Test thoroughly
